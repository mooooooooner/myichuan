## 3. 技术实现总览

### 3.1 鉴权与安全

代理自身鉴权：

- 支持 `Authorization: Bearer <PROXY_API_KEY>`
- 支持 `x-api-key: <PROXY_API_KEY>`

若不匹配返回 401。

### 3.2 上游会话链路

1. 使用 `.env` 的 `SUPABASE_REFRESH_TOKEN + SUPABASE_PUBLISHABLE_KEY`
2. 调用：
   - `POST {SUPABASE_URL}/auth/v1/token?grant_type=refresh_token`
3. 获取并缓存 `access_token`（按 exp 过期前刷新）
4. 使用该 access_token 调用 next-action（`40cd...`）换取 Magai 短 JWT
5. 使用短 JWT 调用：
   - `POST https://beta.magai.co/api/chat`

### 3.3 next-action 请求形态（已按真实 cURL 对齐）

关键点：

- URL：`POST https://beta.magai.co/chat`
- `next-action: 40cd8b2ec4704e0f3c267bd98f93b0f9806e121b77`
- `accept: text/x-component`
- `content-type: text/plain;charset=UTF-8`
- `next-router-state-tree` 固定树
- `cookie: <MAGAI_COOKIE>`
- `apikey: <SUPABASE_PUBLISHABLE_KEY>`
- `authorization: Bearer <Supabase access_token>`
- body：`["<userId>"]`

解析响应中：

- `1:"<short_jwt>"`

### 3.4 自动发现能力

已实现的自动发现：

- `userId`：从 Supabase access_token payload 的 `sub` 解出
- `model 列表`：
  - 从 `rest/v1/spark`（当前用户最近记录）读取 `com_ai_model`，并解析 `chat_json.modelDisplay`
- `chatId`：
  - 优先从历史数据自动发现
  - 若失败可由请求体传入
  - 若仍无则使用 `.env` 的 `MAGAI_DEFAULT_CHAT_ID`

### 3.5 接口协议映射

#### OpenAI 兼容

- 路径：`POST /v1/chat/completions`
- 输入：OpenAI chat 格式（`model/messages/stream`）
- 输出：
  - 非流式：`chat.completion`
  - 流式：SSE + `[DONE]`

#### Claude 兼容

- 路径：`POST /anthropic/v1/messages`（或 `/v1/messages`）
- 输出：
  - 非流式：Anthropic message JSON
  - 流式：`message_start -> content_block_delta -> message_delta -> message_stop`

#### 模型列表

- 路径：`GET /v1/models`
- 内容：动态发现模型（alias + meta）

#### 统计

- 路径：`GET /v1/stats`
- 内容：调用数、错误数、词数统计、平均 TTFT、平均耗时

## 7. 关键注意事项

1. 访问本地接口用 `http://localhost:8787`，不是 `https://`
2. 模型列表接口是 `GET /v1/models`，不是 `/v1/chat/completions`
3. `refresh_token` 可能轮换，建议定期更新 `.env` 中 `SUPABASE_REFRESH_TOKEN`
4. `MAGAI_COOKIE` 过期会导致 next-action 失败
5. 自动发现是 best-effort，生产建议保留 `MAGAI_DEFAULT_CHAT_ID/MODEL_ID` 兜底

## 10. 二次修复记录（2026-05-19）

针对反馈的两个问题，已完成如下修复：

1. `chatId` 需要手动显式配置
- 已修复为自动发现优先：
  - 优先从 `chat` 表最近记录自动拿 `chatId`
  - 其次从 `spark.chat` 回填
  - 再次从 `next-action` 响应兜底提取
  - 最后才使用 `MAGAI_DEFAULT_CHAT_ID`
- 本地实测：`/v1/chat/completions` 不传 `chatId` 可直接返回有效回复。

2. 聊天返回 200 但内容为空
- 根因：上游 `/api/chat` 的 `model` 字段传了展示名（`Claude Sonnet 4.6`），而非真实可执行模型标识。
- 已修复：
  - 模型结构新增 `apiName`（真实请求模型名，如 `anthropic/claude-4.6-sonnet-20260217`）
  - 请求 `/api/chat` 时优先使用 `apiName`
  - `/v1/models` 的 `meta` 增加 `magaiModelApiName`
  - 新增环境变量：`MAGAI_DEFAULT_MODEL_API_NAME`
- 本地实测：已稳定返回文本（例如 `OK`）。

3. refresh token 轮换导致重启失效
- 已修复：refresh 成功后自动持久化最新 `SUPABASE_REFRESH_TOKEN` 到 `.env`，避免“运行时可用、重启后失效”。

4. 运行稳定性修复
- 去掉了 `import.meta` 对 CJS 构建的依赖，`node dist/server.cjs` 可直接运行（不再因 CJS/ESM 差异崩溃）。

### 关于“模型列表不全”的当前结论

- 目前模型发现来源包括：`chat`/`spark` 历史、team action 缓存、可选静态注入 `MAGAI_MODEL_CATALOG_JSON`。
- 若 `.env` 里的 `SUPABASE_REFRESH_TOKEN` 已失效（例如 `refresh_token_already_used`），数据库侧发现能力会显著下降，通常只剩默认模型或历史里已出现的模型。
- 因此要拿到更完整模型列表，建议先更新为最新有效 `SUPABASE_REFRESH_TOKEN`，再调用 `/v1/models` 触发重新发现。
